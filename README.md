# Задание 1. Анализ и планирование

### 1. Описание функциональности монолитного приложения

**Управление отоплением:**

- Пользователи могут регистировать удалять и обновлять новые датчики температуры.
- Пользователи так же могут задавать значение температуры в датчиках, но сейчас в системе нет функционала обновления температуры, более того если после выставления информации об температуре её запросить вернется не то значение которое указал пользователь, а то которое вернет датчик. 

**Мониторинг температуры:**

- Пользователи могут запросить температуру по ID датчика либо по локации
- Система поддерживает запрос данных о температуре из внешнего api.

Поскольку обновления температуры нет, на данном этапе весь проект сводится к api через которое можно загрузить датчик температуры и запросить его показатели

### 2. Анализ архитектуры монолитного приложения

Язык программирования: Go
База данных: PostgreSQL
Архитектура: Монолитная, все компоненты системы (обработка запросов, бизнес-логика, работа с данными) находятся в рамках одного приложения.
Взаимодействие: Синхронное, запросы обрабатываются последовательно.
Развертывание: Требует остановки всего приложения.
Взаимодействие между компонентами: HTTP API 

### 3. Определение доменов и границы контекстов

Реестр устройств - место где можно добавлять, удалять, обновлять информацию об устройствах.
Temperature API - внешний источник данных который по харанее оговоренным правилам возвращает данные о температуре.

### **4. Проблемы монолитного решения**

Текущее монолитное решение не вызывает проблем, основная зависимость которая будет разрастаться и которая не связана с основной системой и так вынесена за её пределы (Temperature API). Для реализации последующих улучшений будет удобнее находиться в одном окружении с реестром устройств, это повысит скорость и надежность разрабоки не нужно будет уточнять изменения в отдельных микросервисах + меньше рисков межсетевого взаимодействия что повышает безопастность (под умные устройства так же могу попасть электронные замки).

# Задание 4. Создание и документирование API

### 1. Тип API

Между микросервисами я использую комбинированный подход: синхронный HTTP/REST + асинхронный обмен сообщениями через брокер (Event Bus)

HTTP/REST - нужен по большей части для микросервисов которые взаимодействют с пользователями
Event Bus - для внутренней логики

Это даёт баланс между простотой и масштабируемостью.
